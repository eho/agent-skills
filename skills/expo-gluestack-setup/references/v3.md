# gluestack-ui v3 Setup

Use the official gluestack CLI as the source of truth for v3 provider/config/component generation. The CLI can require real terminal interactivity, so a user-run checkpoint is a normal part of this workflow, not a failure path.

Do not recreate the v3 provider or components by copying manual docs/source into the app. That manual-copy path has too many moving parts for reliable scaffolding: package metadata, provider source provenance, recursive component dependencies, token config, and CLI config can drift independently. The agent should prepare prerequisites, give exact commands, inspect generated output, repair integration paths, and verify the result.

Track one of these outcomes:

- `user_init_required`: stop and ask the user to run `gluestack-ui init` interactively from the app root.
- `cli_initialized`: init has generated the expected provider/config files and the agent has verified them.
- `user_add_required`: init is verified, starter components are still needed, and the user must run `gluestack-ui add` interactively.
- `components_added`: requested starter components exist and their exports/imports have been verified.
- `blocked`: the user-run CLI command failed, generated output is missing or unusable, prerequisites are incomplete, or verification cannot pass.

## Default Bootstrap

1. Confirm NativeWind is installed and configured before starting gluestack. The gluestack CLI assumes an app that already has the styling pipeline ready.
2. Determine the app root, package manager, route root, global CSS path, and UI component path.
3. Inspect current CLI help only when it can be done non-interactively:

```sh
npx gluestack-ui init --help
```

For Bun:

```sh
bunx gluestack-ui init --help
```

4. Prepare an exact user command for init using documented flags only. Do not guess undocumented flags.
5. Return `user_init_required` and pause before layout wiring, placeholder screens, or component adds.
6. After the user reports init completed, inspect generated files. If provider/config files are present and consistent, set `cli_initialized`; otherwise set `blocked`.
7. If starter components are required, prepare an exact `gluestack-ui add` command, return `user_add_required`, and pause again.
8. After the user reports add completed, inspect generated component folders and exports. If they resolve, set `components_added`; otherwise set `blocked`.

## NativeWind Preflight

Before asking the user to run init, verify:

- The global CSS file used by Metro exists, normally `src/global.css` for SDK 55 `src` layouts.
- That file contains `@tailwind base;`, `@tailwind components;`, and `@tailwind utilities;`.
- `metro.config.js` points `withNativeWind` at the same file, such as `withNativeWind(config, { input: "./src/global.css" })`.
- `tailwind.config.js` uses static `darkMode: "class"` when the provider will use `mode="system"`.
- `darkMode` is not env-driven and cannot resolve to `"media"` under a different shell environment.
- If layout wiring is in scope, the root layout imports the same global CSS file before rendering `GluestackUIProvider`. If layout wiring is not in scope, include the required import in the handoff.

Treat a failed preflight as incomplete setup. Repair it when in scope, or stop with a precise prerequisite before asking the user to run gluestack.

## Init Command

Build the command from the real project shape and confirmed CLI help/docs:

- If `init --help` confirms `--use-bun` and `--path`, use `bunx gluestack-ui init --use-bun --path src/components/ui` for a Bun project with a `src` route/layout.
- If `init --help` confirms `--use-bun` and `--path`, use `bunx gluestack-ui init --use-bun --path components/ui` for a Bun project without `src`.
- If `init --help` confirms `--use-npm` and `--path`, use `npx gluestack-ui init --use-npm --path <ui-path>` for npm projects.
- Use pnpm/yarn package-manager flags only when current help/docs confirm the exact flag names.
- Include an Expo/project-type flag only if current help/docs document the exact flag name.
- If help/docs cannot confirm non-interactive flags, ask the user to run the bare init command with the selected runner and choose the package manager, Expo/React Native project type, and UI path interactively.

When pausing with `user_init_required`, tell the user:

- The exact working directory.
- The exact command to run.
- The selected package manager and UI component path.
- The prompt choices to preserve if the CLI asks, especially Expo/React Native project type, package manager, and component path.
- To reply when the command has finished, including any error output if it fails.

Do not continue to `gluestack-ui add`, theme wiring, placeholder screens, or full scaffold verification until the user-run init has been inspected.

## Post-Init Verification

After the user reports init completed:

- Inspect the selected UI directory for `gluestack-ui-provider` and its exported `GluestackUIProvider`.
- Inspect generated CLI config, normally `gluestack-ui.config.json`, when present.
- Inspect `package.json` and lockfiles for gluestack-added packages and unexpected package-manager drift.
- Confirm Tailwind content globs include the actual app, src, and component directories.
- Confirm Metro still points to the intended NativeWind CSS file.
- Confirm `tailwind.config.js` still uses static `darkMode: "class"` for provider mode `"system"`.
- Identify the provider import string and provider mode for the orchestrator.

Set `cli_initialized` only when the provider/config output is real and consistent. If init failed, left only partial output, changed the wrong files, or produced a package-manager/lockfile state the agent cannot reconcile safely, set `blocked`.

## Starter Components

Use CLI-managed components only. Do not copy component source from docs or GitHub as a fallback.

Recommended scaffold starter set:

```sh
box vstack hstack text heading button card input badge divider icon toast alert
```

Before asking for add, inspect `gluestack-ui add --help` when possible:

```sh
bunx gluestack-ui add --help
```

or:

```sh
npx gluestack-ui add --help
```

If multi-add is documented, ask the user to run:

```sh
bunx gluestack-ui add box vstack hstack text heading button card input badge divider icon toast alert
```

Use `npx gluestack-ui add ...` for npm projects, and the matching runner for pnpm/yarn.

If multi-add is not documented, ask the user to add components one at a time with the same runner. Preserve the intent of the starter set when component names change according to CLI suggestions:

- Layout: Box, VStack, HStack
- Typography: Text, Heading
- Action: Button
- Form: Input
- Display: Card, Badge, Divider
- Feedback: Toast or Alert
- Media/icon: Icon

When pausing with `user_add_required`, tell the user:

- The exact working directory.
- The exact command or ordered command list.
- That the project must already contain the verified init output.
- To reply with any CLI errors if add fails.

Do not wire a placeholder screen that imports components until the user-run add has been inspected.

## Post-Add Verification

After the user reports add completed:

- Confirm each required component folder exists in the selected UI path.
- Confirm the component exports that the scaffold will import.
- Run TypeScript or the project’s normal static check when available.
- Verify no add command reported `gluestack is not initialized in the project`.
- Confirm package-manager and lockfile state still match the selected package manager.

Set `components_added` only when the component files and exports are verified. If a component failed to add and no smaller official CLI starter set can be verified, set `blocked`.

## Lockfile Hygiene

After each user-run gluestack command, inspect the app directory for unexpected lockfiles. The CLI may call npm internally and create `package-lock.json`. If the project uses Bun, remove `package-lock.json`, `yarn.lock`, and `pnpm-lock.yaml` only when they are clearly generated by the gluestack command and the user did not request them, then run the chosen package manager install again if needed.

Do not remove workspace-local `node_modules` link folders that Bun created solely because a root lockfile exists. They may contain links/shims required by package-local scripts.

## Provider Wiring

After `cli_initialized`, identify how the generated provider should be wired. In standalone mode, perform the wiring when it is in scope. In orchestrated mode, report the wiring details without editing layout files unless the caller explicitly allowed that.

For Expo Router, the provider normally belongs in `src/app/_layout.tsx` around the root `Stack` for SDK 55-style projects, or `app/_layout.tsx` for templates that still use root-level routes. Ensure the global CSS import appears before rendering UI.

For the default scaffold, pass `mode="system"` to `GluestackUIProvider` when the generated provider supports it.

## Post-CLI Config Repair

gluestack CLI may rewrite app config files. Immediately inspect and repair gluestack-owned integration points when in scope:

- `metro.config.js`: NativeWind input path must match the real CSS file.
- `global.css`: the Metro input CSS file must contain the Tailwind base/components/utilities directives.
- `babel.config.js`: preserve existing plugins and aliases.
- `tailwind.config.js`: content globs must match the real app, src, components, and UI paths. For provider mode `"system"`, `darkMode` must be static `"class"`.
- `src/app/_layout.tsx` or `app/_layout.tsx`: keep exactly one global CSS import and one provider wrapper when layout wiring is in scope. Otherwise report the required layout import/wrapper details in the handoff.
- TypeScript aliases: ensure `@/*` and `@/assets/*` resolve consistently with Metro and Babel.

## Placeholder Screen

When the caller provides a starter screen example, use it only as a structure and adjust imports to match verified CLI-generated component paths. If a component API differs, prefer the generated component API. In orchestrated mode, do not edit placeholder routes unless explicitly allowed; return component export names and import paths for the orchestrator.

## Verification Criteria

Before reporting gluestack success, verify:

- The setup outcome is `cli_initialized` when no starter components are required, or `components_added` when starter components are required.
- The generated provider exists and exports `GluestackUIProvider`.
- `gluestack-ui.config.json` exists when the current CLI generates/requires it.
- NativeWind preflight still passes for provider system mode.
- Starter component files exist at the selected UI path when components were requested.
- Placeholder screen imports use verified CLI-generated component paths when route/screen edits were in scope.
- Any package or lockfile repair is documented in the handoff.

If the outcome is `user_init_required`, `user_add_required`, or `blocked`, do not describe setup as complete.
